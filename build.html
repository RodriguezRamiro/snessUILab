SNES UI Lab — System Build Documentation

This document describes the internal architecture of the SNES UI Lab runtime. It outlines how the engine operates, how games integrate into the system, and how core systems interact during execution.

Engine Overview

SNES UI Lab is a modular browser-based console runtime built using JavaScript and HTML5 canvas rendering. The system is designed to simulate a cartridge-based console environment where individual games operate as isolated modules connected to a shared runtime engine.

  • Fixed-timestep update loop
  • Canvas-based rendering pipeline
  • Modular game cartridge loading
  • Deterministic execution behavior
  • Shared runtime state management

Runtime Execution Flow

The engine operates using a continuous frame loop driven by requestAnimationFrame. Each frame processes input, updates game logic, and renders the current state to the screen. 

Engine Loop
    ↓
Input System
    ↓
Game Logic Update
    ↓
Collision Detection
    ↓
State Management
    ↓
Rendering Pipeline

This predictable execution order ensures consistent physics behavior, reliable input handling, and stable rendering output.

Core Runtime Systems

The runtime includes several shared systems that support all games running inside the console environment.

  • Score tracking system
  • Pause state handling
  • Game over signaling
  • Shared system memory
  • Input state monitoring
  • Frame timing control

These systems operate independently of individual games, allowing new cartridges to be added without modifying the core engine.

Cartridge Architecture

Each game in the SNES UI Lab environment functions as a self-contained module referred to as a cartridge. Cartridges interact with the console through a standardized interface.

Required Cartridge Interface

GameCartridge

Properties:
- name
- version

Methods:
- init()
- update(deltaTime)
- render(context)
- destroy()

The Cartridge Manager is responsible for loading, switching, and unloading games while preserving system stability.

  • Game initialization
  • Active game switching
  • Resource cleanup
  • Memory isolation

Physics and Collision Handling

Movement and collision detection are processed during the update phase of the engine loop. The physics system calculates object motion using velocity, acceleration, and positional updates per frame.

  • Position updates based on velocity
  • Boundary collision detection
  • Ground and horizon detection
  • Crash state triggering

Collision logic is executed before rendering to ensure accurate visual feedback and consistent gameplay behavior.

Rendering Pipeline

The rendering system is responsible for drawing all visual elements to the screen using the HTML5 Canvas API.

  • Screen clearing
  • Background rendering
  • Object drawing
  • HUD display updates
  • Frame presentation

Rendering occurs after all physics and game logic updates have been completed for the current frame.

Runtime Status

The console runtime is currently stable and operational. The system supports modular expansion and predictable execution behavior across multiple game cartridges.

New games can be integrated into the environment without modifying the core engine architecture.
